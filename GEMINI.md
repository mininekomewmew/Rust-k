# Project Engineering Standards

## 1. NO MAGIC
Avoid opaque or non-obvious logic. Favour explicit implementation and readable code over clever hacks or hidden state.

## 2. VERIFY BEFORE DONE
Never claim a task is complete without empirical evidence. Always run tests, check build logs, or perform manual verification before reporting "DONE".

## 3. DISSENT
Challenge assumptions. If a proposed solution seems flawed, inefficient, or deviates from established patterns, raise the concern immediately.

## 4. SCOPE DRIFT
Strictly adhere to the agreed-upon task scope. Do not perform unrelated refactoring or add features not explicitly requested in the implementation plan.

## 5. EXPLICIT ASSUMPTIONS
Clearly state any assumptions made during research, design, or implementation. Document "Why" a specific path was chosen when alternatives exist.
